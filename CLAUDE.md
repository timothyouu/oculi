# Project: Oculi

## Purpose
Oculi is a photo-spot discovery app: a map-first Next.js experience for finding, saving, and route-planning photography locations. Users browse places on a live Mapbox map, filter/search them, save favorites, and build shoot routes.

## Stack
- Frontend: Next.js 14 (App Router), React 18 (functional components + hooks), Tailwind CSS
- Backend: Next.js route handlers (`app/api/*`)
- Database: Supabase (`@supabase/supabase-js`, `lib/supabase.ts`)
- Maps: mapbox-gl v3 (live map) with a stylized SVG fallback (`components/stylized-map.tsx`)
- AI/ML: none currently
- Testing: Vitest (`*.test.ts` unit tests in `lib/`), Playwright (`e2e/`)

## Project Structure
- `app/` — App Router routes: `map/`, `places/`, `profile/`, `saved/`, plus `api/` route handlers (incl. `api/mapbox` proxy) and root `layout.tsx`/`page.tsx`.
- `components/` — React UI. Key: `mapbox-map.tsx` (live map + fallback orchestration), `stylized-map.tsx` (no-token/auth-failure fallback), `saved-panel.tsx` (saved-places grid + the user-built **Itinerary** panel), `place-detail.tsx`, `selected-place-card.tsx`, `share-place-button.tsx` (single-click copy-to-clipboard share button, used on both).
- `lib/` — pure logic + data: `map-clusters.ts` (marker clustering), `mapbox-fallback.ts` (fallback decision), `place-taxonomy.ts` (canonical scene + best-light vocab, and the pure `matchesSceneFilter`/`matchesBestLightFilter` predicates + `sceneTypesForPlace`/`bestLightForPlace`/`sceneLabelsFor` accessors the map filters and place UI share), `place-accessibility.ts` (`accessibilityForPlace` returns the curated `place.easeOfVisit`, deriving Easy/Moderate/Difficult from tags/times/description only as a fallback), `search-ranking.ts`, `geo.ts` (haversine distance + nearest-first sorting, used by the add-photo location typeahead), `share-place.ts` (`buildPlaceShareUrl` — the only share helper left after the share popover was replaced with a direct-copy button), `discovery-queue.ts` (`buildResumableQueue` — pure resume-queue logic for the Discover deck, see Architecture Notes), `catalog-validation.ts` (`parsePhotoPayload`/`parsePlacePayload`/`parseAreaPayload`/`parseUserPayload` — defensive per-kind validators for Supabase-hydrated catalog/photo rows, see Architecture Notes), `itinerary.ts` (`resolveItineraryPlaces`/`buildItinerary`/`itineraryGoogleMapsUrl`/`itineraryAppleMapsUrl` — pure helpers for the user-built saved itinerary; replaced the auto-generated `saved-route-planner.ts`), `storage.ts`, `data.ts`, `types.ts`, `supabase.ts`. Unit tests live beside their modules as `*.test.ts`.
- `e2e/` — Playwright specs. `supabase/` — Supabase config. `design/`, `spec.md`, `specs.md`, `implementation.md` — planning docs.

## Architecture Notes
- **Mapbox token**: read client-side from `NEXT_PUBLIC_MAPBOX_ACCESS_TOKEN`. Without it, the component renders `StylizedMap`.
- **Mapbox proxy** (`app/api/mapbox/route.ts`): Mapbox API requests are rewritten (via `transformRequest` in `mapbox-map.tsx`) through `/api/mapbox`, which forwards them to `api.mapbox.com` with a controlled `Referer` (and, currently, `Origin`) so referrer-restricted tokens work server-side. It retries with a `http://localhost:3000/` referrer on 401/403 in local dev.
- **Live-map vs. fallback decision** is isolated in `lib/mapbox-fallback.ts` (`shouldFallbackToStylizedMap`). The live map is only abandoned when there is no token or Mapbox returns a confirmed auth rejection (401/403). Transient Mapbox GL `error` events (canceled tiles, sprite hiccups) must NOT tear the map down — doing so was a regression fixed on 2026-07-08.
- **Markers/nodes** only mount on the live map once `mapReady` is true; they are clustered by zoom via `lib/map-clusters.ts`.
- **Map camera position persists across sessions** (`lib/storage.ts` `loadMapCameraView`/`saveMapCameraView`, key `oculi:map-camera`). `components/mapbox-map.tsx` saves `{center, zoom, bearing, pitch}` on every `moveend`, and on first mount `jumpTo`s (instant, no animation) straight to the saved view instead of running the "fit to all places" `fitBounds` — gated by `hasRestoredCameraRef` so later re-fits (e.g. a filter narrowing the visible places) still behave normally. A one-shot `suppressNextAutoFocusRef` prevents the separate `autoFocusSelected` effect (which re-centers on a restored/searched selected place) from immediately panning away from the just-restored camera on that same mount; it only suppresses the first firing, so a live search still auto-focuses normally. The stylized (no-token) fallback has no real camera, only a cluster "detail level" stepper, so `components/stylized-map.tsx` persists that instead (`loadMapDetailLevel`/`saveMapDetailLevel`, key `oculi:map-detail-level`) for parity.
- **Cluster clustering is screen-pixel-based, not geographic** (`lib/map-clusters.ts` `clusterProjectedPlacePhotoNodes` groups by projected on-screen distance). At low zoom (e.g. the initial world view) this means a single "cluster" can contain places spread across continents — their true `LngLatBounds` is then nearly the whole visible world already. Clicking such a cluster in `components/mapbox-map.tsx` computes `map.cameraForBounds(...)` first and only calls `fitBounds` if that would actually zoom in meaningfully; otherwise it falls back to a flat zoom step (`currentZoom + 3`, capped at 16) centered on the cluster so every click makes visible progress. Fixed 2026-07-09 — see below.
- **The camera-fit-to-all-places effect must key off the visible place *id set*, not the `places` array reference.** `components/mapbox-map.tsx`'s effect that re-fits the camera to the whole `places` list depends on `visiblePlaceIdsKey` (sorted, joined place ids) rather than `places` directly. Selecting any place calls `recordPlaceView` (`app/map/page.tsx`), which bumps `recentActivityScore` and re-sorts `topPlaces` upstream — a new array reference with the same members. If this effect depended on `places` directly, every single place selection would re-trigger a "fit to everything" `fitBounds` call that cancels any in-flight camera animation (e.g. a cluster-zoom just triggered by that same click) and snaps the view back out. Do not revert this to depend on `places` directly.
- **The Discover deck's "resume where you left off" queue must always keep the resume anchor in the queue, even though it was just marked viewed.** `components/discover-deck.tsx` builds its swipe queue by filtering `allCards` down to unseen photos (`viewedPhotoIds` exclusion, so already-swiped cards don't reappear forever). But landing on a card immediately records it as viewed (the `onViewPhoto` effect), so a naive "unseen only" filter drops the exact card the user was just looking at from the very next queue rebuild — the persisted `lastDiscoveryPlaceId`/`discoveryActiveIndex` resume target then isn't found in the rebuilt queue, `findIndex` returns -1, and the deck falls back to a numeric index into a reshuffled, shorter queue — landing on a different, effectively arbitrary place every reload. Fixed 2026-07-09 by extracting `lib/discovery-queue.ts` `buildResumableQueue(allPhotoIds, viewedPhotoIds, resumePhotoId)`, a pure/unit-tested helper that filters to unseen photos but always keeps `resumePhotoId` in the result (in its natural position) regardless of viewed status. `discover-deck.tsx` computes the resume anchor's photo id from `resumePlaceId` and passes it through on every queue-build. Any future change to the queue-building filter must preserve this invariant — don't let a plain "exclude viewed" filter creep back in.
- **Supabase migration numbering**: the remote project (`xlzknvhiuhtcqmqrypqh`) records applied migrations in `supabase_migrations.schema_migrations` by *version number*, and the Supabase GitHub "Preview" check fails with "Remote migration versions not found in local migrations directory" if a remote version has no matching filename in `supabase/migrations/`. Do NOT renumber/rename a migration file after it has been applied remotely. On 2026-07-08, `saved_route_planner_support` was applied remotely as `20260708025150` but renumbered in-repo to `20260708000300`; the fix was realigning remote history (`update ... set version='20260708000300' where version='20260708025150'`) rather than editing files.

- **Place taxonomy is stored, not inferred** (added 2026-07-09). Every `Place` carries three curated fields alongside `tags`/`bestTimes`: `sceneTypes: string[]` (subset of the eight canonical scene values in `lib/place-taxonomy.ts` `SCENE_VALUES`), `easeOfVisit: "Easy"|"Moderate"|"Difficult"`, and `bestLight: string[]` (subset of the six `LIGHT_VALUES`). The map's Scene type / Ease of visit / Best light filters (`app/map/page.tsx`) match these fields **exactly** (`matchesSceneFilter`, `accessibilityForPlace`, `matchesBestLightFilter`) — they no longer substring-match tags or alias-match `bestTimes`. `bestTimes` remains free-form display flavor; `bestLight` is the filter-grade field. When adding/editing a place you MUST set all three, keep `sceneTypes ⊆ SCENE_VALUES` and `bestLight ⊆ LIGHT_VALUES` (the `lib/place-taxonomy.test.ts` integrity test enforces non-empty subsets over the whole catalog), and mirror the change into Supabase (see below). The taxonomy helpers keep a tags/bestTimes-derived fallback only for rows that predate the fields — don't rely on it for seed data.
- **Regenerating the place catalog migration**: `lib/data.ts` is the source of truth. After editing places, run `npx tsx scripts/gen-taxonomy-migration.mts` to rewrite `supabase/migrations/20260709000100_place_taxonomy_fields.sql` (upserts all 61 full `Place` payloads as jsonb into `oculi_demo_catalog_items`), then apply it to the remote project (`xlzknvhiuhtcqmqrypqh`). `scripts/apply-taxonomy.mjs` is the one-shot that first injected the fields into `data.ts` (idempotent — skips places that already have `sceneTypes`).

- **`SelectedPlaceCard` (the map's floating place card) has three internal views, not two**: `"overview"` (place info + Save/Share), `"gallery"` (list of that place's photos, opened via the "N photos" pill over the cover image), and `"post"` (a single photo's full detail — added 2026-07-09). Clicking a photo row in the gallery sets `selectedPhotoIndex` and switches to `"post"`, which shows the full image with prev/next chevrons (wrapping within that place's `visiblePhotos`), the photographer (`users` prop, looked up via `usersById` by `photo.userId`), caption, like count, real date, and tags. Every view has its own header with a back arrow (steps up one level: post→gallery→overview) **and**, wherever `onClose` is provided, an `X` that closes the whole card straight back to the map from any depth — don't remove the `X` from the gallery/post headers thinking the arrow alone is sufficient; users expect a one-click way out at every level, not just from `overview`.
- **`SelectedPlaceCard` needs `users` threaded all the way from `app/map/page.tsx`**: `useDemoState()` → `<MapboxMap users={users} .../>` → (live map path) `<SelectedPlaceCard users={users} .../>` and (fallback path) `<StylizedMap users={users} .../>` → `<SelectedPlaceCard users={users} .../>`. If you add a new field that needs photographer data in this card, it's already wired — just read `activePhotographer`/`usersById` in `selected-place-card.tsx`. `users` defaults to `[]` at every layer so the prop is optional and nothing crashes if a caller forgets it (photographer info just won't render).
- **`SelectedPlaceCard`'s gallery/post like button now does real per-photo liking, threaded the same way `users` is** (added 2026-07-09): `likedPhotoIds`/`onTogglePhotoLike` flow `useDemoState()` → `app/map/page.tsx`'s `<MapboxMap likedPhotoIds={state.likedPhotoIds} onTogglePhotoLike={togglePhotoLike} .../>` → both `mapbox-map.tsx` and `stylized-map.tsx` → `<SelectedPlaceCard likedPhotoIds={...} onTogglePhotoLike={...} .../>`, both defaulting to `[]`/undefined so the prop is optional. The gallery row is no longer a single `<button>` wrapping both the photo and a decorative like chip (that chip had no click handler of its own and was previously indistinguishable from the real "Like photo" pill used in `photo-card.tsx`, so clicking it silently opened the post instead of liking anything — a genuine misleading-button bug, not just a11y nitpick). It's now a `<div>` row containing two sibling `<button>`s: one for "open this post" (image + caption + photographer) and one real, independently-clickable Like toggle (mirrors `photo-card.tsx`'s pill styling/behavior). The post-detail view's like count was similarly upgraded from a static `<span>` to a real toggle button. Any future gallery/list row that pairs a "navigate" action with a "like/save" stat must follow this sibling-buttons pattern — never nest an interactive-looking element inside another `<button>`, since the outer click handler always wins and the inner one becomes dead decoration.
- **Icon conventions must stay app-wide, not per-component**: `Heart` always means "like a photo" and `Bookmark` always means "save a place" everywhere in the app (`photo-card.tsx`, `place-card.tsx`, `selected-place-card.tsx`, `place-detail.tsx`). `components/discover-deck.tsx`'s swipe-right "save place" action previously used a `Heart` icon (both in the on-card button and its own "Swipe right to save" tutorial hint), which broke that convention — fixed 2026-07-09 by switching both to `Bookmark`. Don't introduce a new save/like affordance anywhere without checking which of the two it actually is and using the matching icon.
- **`place-detail.tsx`'s third action-row icon button is "Add to shoot-day route", not a "more actions" menu** (fixed 2026-07-09, was previously a `MoreHorizontal` "…" icon labeled `Show more actions for {place}` whose `onClick` just set a fake status string — "Added to your next shoot-day options." — without touching any real state; the shoot-day route planner on `/saved` (`saved-panel.tsx`) only ever pulls stops from already-saved places, so the old button's claimed action was completely fictitious). It's now a `Route`-icon button that actually saves the place (via `onToggleSaved`, only if not already saved — never re-toggles an already-saved place, which would unsave it) so it truly becomes eligible for the shoot-day planner, with `aria-label`/status text that describe that real behavior. Don't reintroduce an overflow/"more actions" icon here unless it opens an actual menu with actual actions in it.

- **Cluster clicks zoom, they never select** (fixed 2026-07-12). In both `components/mapbox-map.tsx` and `components/stylized-map.tsx`, clicking a multi-place cluster marker only zooms in (fitBounds/flat zoom step on the live map, detail-level bump on the stylized fallback) — `onSelectPlace` fires ONLY in the lone-marker (`cluster.places.length === 1`) branch, so the `SelectedPlaceCard` opens only when the user clicks an actual single-place node. Marker aria-labels reflect this: multi-place clusters are labeled `Zoom into …`, lone markers `Select …` — e2e specs (`e2e/map-camera.spec.ts`, `e2e/share-place.spec.ts`) key off these prefixes, and `map-camera.spec.ts` asserts no card appears after a cluster click. Don't move `onSelectPlace` back outside the lone-marker branch or unify the labels.
- **`saved-panel.tsx`'s saved-place cards have exactly one bookmark icon, next to the title, and unsaving requires confirmation** (fixed 2026-07-12). The cover photo previously carried its own gold bookmark ribbon overlay in addition to the bookmark button next to the title — a redundant second affordance, removed so the cover image is purely a click-to-open-place target. Clicking the remaining bookmark button no longer calls `onToggleSaved` directly; it sets `placePendingRemoval` (the `Place` object), which renders `ConfirmRemoveSavedPlaceDialog` — a `role="dialog"` modal matching the style already used in `profile-summary.tsx` — with Cancel/Remove actions. Only Remove calls `onToggleSaved`; Cancel or a click on the backdrop just clears `placePendingRemoval`. Don't reintroduce a second bookmark affordance on the cover photo, and don't wire the bookmark button back to an instant unsave — any future "remove saved place" entry point should route through this same confirm-first pattern.
- **The `/saved` aside is a user-built Itinerary, not an auto-generated shoot-route planner** (replaced 2026-07-12). There is no morning/sunset (sunrise/fog vs. golden hour) route generation, no route tabs, no pin/regenerate/save-route-plan, no arrival/duration flavor — all of that (`lib/saved-route-planner.ts`, `lib/remote-route-plans.ts`, and the `route_plans`/`route_plan_stops` write path) was deleted. `components/saved-panel.tsx`'s `ItineraryPanel` is a plain ordered list the user fills themselves, two ways: (1) an **Add** button (`aria-label="Add saved place to itinerary"`) opens `ItineraryAddDialog` (`role="dialog"`, aria-label "Add saved places to itinerary") listing saved places not yet in the itinerary, each a `Add {name} to itinerary` button; (2) a three-dots **More actions** menu on each saved-place card (`aria-label="More actions for {name}"`, a real `role="menu"` with an "Add to itinerary"/"Remove from itinerary" `menuitem` plus "Open place" — this is a legit menu-with-actions, not the fictional "…" pattern banned elsewhere). Stops are `[data-itinerary-stop-id]` rows, drag-reorderable (same pointer/mouse hit-test pattern the route planner used), each with a `Remove {name} from itinerary` X and Copy/Google/Apple handoffs; the panel's Google/Apple buttons build a whole-route directions URL via `itineraryGoogleMapsUrl`/`itineraryAppleMapsUrl`. **Persistence:** the itinerary is `DemoState.itineraryPlaceIds` (an ordered id list) kept in the `oculi_demo_states` JSON blob — it is NOT one of the four relation tables, so it rides the debounced whole-state save like `viewedPlaceIds`/`placeViews` (actions `addToItinerary`/`removeFromItinerary`/`reorderItinerary` in `lib/demo-state.tsx` just `update` state; do NOT route it through the granular `saved_places`-style remote helpers). `resolveItineraryPlaces` resolves ids against the FULL catalog (not just saved places) so a stop survives being un-saved, while the add picker only ever offers currently-saved places. Don't reintroduce auto route generation or the light-window flavor here — if smart routing is ever wanted again it's a new feature, not a revert.

## Active Constraints
- Always run the local dev server / test the app on `localhost:3000` specifically — the Mapbox token/referrer setup (`app/api/mapbox/route.ts`, see Architecture Notes) is tied to that origin, so a different port (e.g. Next.js auto-picking 3001 because 3000 is in use) will break live Mapbox auth even though everything else works. If 3000 is occupied, free it rather than letting the dev server move to another port.
- Do not couple transient Mapbox errors to the fallback. Only 401/403 (tracked via `mapUnauthorized`) should flip to `StylizedMap`; other errors are cosmetic banners over the live map.
- Keep map/fallback decision logic pure and in `lib/` so it stays unit-testable without importing `mapbox-gl`.
- Functional components + hooks only. TypeScript strict; type all signatures.
- On `/map`, `selectedPlaceId` must never default/fallback to `mapPlaces[0]`/`places[0]` — the place card (`SelectedPlaceCard`) only shows after an explicit selection (node click, search match, restored session, or "Near me"). Both `MapboxMap` and `StylizedMap` intentionally do `places.find(...)` with no `|| places[0]` fallback; don't reintroduce one.
- The map card (`selected-place-card.tsx`) has exactly one save affordance: the labeled Save/Saved button in the action row. Do not reintroduce a second bare bookmark-icon toggle (e.g. a corner icon over the cover photo) — that was removed 2026-07-09 as a deliberate design call (the labeled button reads as more intuitive/discoverable than an icon-only toggle).
- Sharing a place is a single click that copies the place URL to the clipboard (`components/share-place-button.tsx`) — no dropdown/popover with social targets or native-share menu. Don't reintroduce a share menu; if a future request wants social targets back, that's a new design decision, not a revert.
- The cover-photo bookmark ribbon on `place-detail.tsx` (the gold badge in the top-right corner of the hero photo) must only render when `isSaved` is true. It was previously unconditional decoration; fixed 2026-07-09. Every other `Bookmark` icon in the app (`place-card.tsx`, `photo-card.tsx`, `selected-place-card.tsx`) was already correctly gated on real saved state, and `saved-panel.tsx`/`profile-summary.tsx`'s badges are safe unconditional because those lists only ever contain saved places by construction — don't add a new unconditional bookmark badge without checking which case it falls into.
- `SharePlaceButton` has no built-in visual feedback beyond swapping its own icon to a checkmark — it reports status via an `onStatusChange?: (message: string | null) => void` callback, and the caller renders that message in whatever inline notice slot it already has (`selected-place-card.tsx`'s `cardNotice` line, `place-detail.tsx`'s `actionStatus` line). An earlier version rendered its own absolutely-positioned tooltip below the button; that got removed 2026-07-09 because the popup/card container measures its content's bounding box (including absolutely-positioned descendants), so the "floating" tooltip was visually stretching the card taller instead of overlaying it. Any new `SharePlaceButton` usage must wire `onStatusChange` into an existing in-flow status line rather than adding a new floating tooltip.

2026-07-09 — Fixed map camera transitions jump-cutting instantly instead of animating on the deployed site, even though local/Playwright testing showed them working. Root cause: Mapbox GL JS silently forces `duration: 0` (skips the eased animation, jumps straight to the end camera position) whenever the browser reports `prefers-reduced-motion: reduce`, *unless* the call passes `essential: true` — none of `components/mapbox-map.tsx`'s camera calls did. Playwright's default browser context doesn't set that media feature, so every prior Playwright verification of the cluster/lone-marker zoom (this file, entries below) looked correct locally while a real deployed browser with reduced-motion on (OS setting or otherwise) would see an instant cut. Reproduced by using Playwright's `page.emulateMedia({ reducedMotion: 'reduce' })` (a real browser-level media-feature override, not a JS shim — a shim doesn't work because Mapbox caches the `MediaQueryList` from its first `matchMedia` call and reads `.matches` live off that object) against the still-deployed pre-fix build (`oculi-demo.vercel.app`): the camera jumped from the world view straight to the fully-zoomed North America view within 2 animation frames (~32ms) of the click. Added `essential: true` to all four camera calls in the file (`fitPlaces`'s single- and multi-place branches, both branches of the marker click handler, and the `autoFocusSelected` effect). Verified the fix locally with the same `emulateMedia` reduced-motion override: the same click now visibly progresses over ~200-650ms instead of snapping instantly. **Constraint:** any new `map.easeTo`/`flyTo`/`fitBounds` call added to this file must include `essential: true`, or it will silently skip animation for any user (or automated test) with reduced-motion enabled — Playwright will not catch a missing `essential: true` under default settings, so don't rely on visual Playwright verification alone for camera-animation changes; either force-check with `emulateMedia` or inspect the call site.
